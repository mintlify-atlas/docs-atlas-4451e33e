---
title: Generating operations
description: Regenerate AzuraCast operations from the official OpenAPI source
---

All node operations, properties, and execution handlers are generated from the official AzuraCast OpenAPI specification. When AzuraCast updates its API ‚Äî adding, removing, or modifying endpoints ‚Äî you regenerate to keep the node in sync.

## OpenAPI source

The generator fetches the spec from:

```
https://raw.githubusercontent.com/AzuraCast/AzuraCast/main/web/static/openapi.yml
```

You can override this at runtime with the `AZURACAST_OPENAPI_URL` environment variable if you need to point at a different revision or a local file.

## Regeneration flow

<Steps>
  <Step title="Fetch OpenAPI and generate snapshot">
    ```bash
    npm run generate:operations
    ```

    This script does two things in sequence:

    1. Fetches `openapi.yml` from the AzuraCast source URL
    2. Parses all path items and extracts every operation ID, method, path, tag, parameters, request body schema, and response content types
    3. Writes `nodes/AzuraCast/azuracast.openapi.snapshot.json` with the full operation list, a keyed `operationMap`, and a flat `operationOptions` array sorted by display name
    4. Runs `generate:nodes` automatically, which reads the snapshot and regenerates all files in `properties/` and `execute/`, plus `azuracast.domains.json`
  </Step>

  <Step title="Verify operation coverage">
    ```bash
    npm run verify:operations
    ```

    This re-fetches the live OpenAPI spec and compares it against the snapshot. It checks for:

    - Missing operations (in live spec but not in snapshot)
    - Extra operations (in snapshot but not in live spec)
    - Metadata drift (method, path, tag, or public flag changed)
    - Domain manifest completeness (all operations assigned to a domain)
    - Generated file counts (one `*.operations.ts`, `*.fields.ts`, and `execute/*.ts` per domain)
    - `package.json` node registration

    A passing run prints:

    ```
    Operation coverage verified: 263/263 across N grouped resources in one node (metadata aligned)
    ```

    If any check fails, the script exits with code 1 and prints the specific issues to stderr.
  </Step>

  <Step title="Verify UI metadata">
    ```bash
    npm run verify:ui
    ```

    <Warning>
      This step requires a successful `npm run build` first. It imports from `dist/`.
    </Warning>

    This loads the compiled `properties/index.js` and checks that every resource and action option renders correctly in the n8n UI:

    - Resource selector has options and a valid default
    - Each resource has exactly one operation selector
    - Every operation option has a non-empty name, value, and action
    - No operation label looks like a URL path or contains brackets
    - No operation label exceeds 8 words
    - No duplicate labels or values exist

    A passing run prints:

    ```
    UI metadata verified: resources=N operationSelectors=N operations=263
    ```
  </Step>

  <Step title="Build">
    ```bash
    npm run build
    ```

    Compiles the regenerated TypeScript to `dist/`. Run this after generation to confirm the output compiles cleanly.
  </Step>
</Steps>

## Generated files

### `azuracast.openapi.snapshot.json`

Stored at `nodes/AzuraCast/azuracast.openapi.snapshot.json`. This file is the source of truth for the runtime. It contains:

- `source` ‚Äî the OpenAPI URL and generation timestamp
- `operations` ‚Äî full array of all parsed operations, each with:
  - `id`, `method`, `path`, `tag`, `summary`, `description`
  - `pathParameters`, `queryParameters` (name arrays)
  - `pathParameterDefinitions`, `queryParameterDefinitions`, `requestBodyFieldDefinitions` (typed field definitions)
  - `requestBodyContentTypes`, `requestBodyPreferredContentType`, `recommendedBodyMode`
  - `responseContentTypes`, `recommendedResponseFormat`
  - `isPublic` flag
- `operationMap` ‚Äî operations keyed by operation ID for O(1) lookup at runtime
- `operationOptions` ‚Äî flat array of `{ name, value, description }` objects sorted alphabetically for display

### `azuracast.domains.json`

Stored at `nodes/AzuraCast/azuracast.domains.json`. This is the domain groupings manifest. It records every API tag as a domain with:

- `tag` ‚Äî the official AzuraCast API tag (e.g. `Stations: Media`, `Administration: Users`)
- `resourceValue` ‚Äî the sanitized value used in the n8n `resource` parameter
- `displayName` ‚Äî the human-readable label shown in the resource selector
- `operationCount` and `operationIds` ‚Äî the operations assigned to this domain

`verify:operations` cross-references this file against the live OpenAPI spec to detect domain drift.

### `properties/` directory

Generated by `generate:nodes`. Contains one pair of files per domain:

- `<domain>.operations.ts` ‚Äî exports a single `INodeProperties` object that is the operation selector for that resource, displayed when the user selects that resource in the n8n UI
- `<domain>.fields.ts` ‚Äî exports an array of `INodeProperties` objects, one per field (path parameter, query parameter, or request body field), each gated behind a `displayOptions.show` condition matching the operation

All per-domain exports are collected in `properties/index.ts` as `azuraCastNodeProperties`, which is what `AzuraCast.node.ts` uses.

### `execute/` directory

Generated by `generate:nodes`. Contains one file per domain:

- `<domain>.ts` ‚Äî exports a record mapping operation IDs to executor functions; each executor calls `executeOperationById` from `AzuraCast.shared.ts`

All per-domain records are collected in `execute/index.ts` as `resourceOperationsFunctions`, which `AzuraCast.node.ts` dispatches into at runtime.

## What happens when coverage drops

If the AzuraCast API adds new operations that are not in the snapshot, `verify:operations` will report them as missing and exit with code 1. The `validate:publish` script runs `verify:operations` as part of its pipeline, so publishing will be blocked until you regenerate.

To recover:

```bash
npm run generate:operations
npm run verify:operations
npm run build
npm run verify:ui
```

<Info>
Operations are identified by `operationId` from the OpenAPI spec. If AzuraCast renames an operation ID, `verify:operations` will report the old ID as extra and the new one as missing. After regeneration, existing n8n workflows that reference the old operation ID by value will still work as long as the `operationId` string is preserved ‚Äî this is the backward compatibility path documented in `AzuraCast.shared.ts`.
</Info>
